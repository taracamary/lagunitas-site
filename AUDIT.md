# Технический аудит и целевая архитектура

## ШАГ 1 — Глубокий технический аудит

### 1. HTML (Семантика, a11y, структура)
*   **Отсутствие семантики:** Весь контент обернут в безликие `div` (`#page1`, `#page2` и т.д.). Нет тегов `<header>`, `<main>`, `<section>`, `<article>`, `<footer>`. Это плохо для SEO и скринридеров.
*   **Провал по доступности (a11y / WCAG):** 
    *   Абсолютно все изображения имеют пустой атрибут `alt=""`. Скринридеры просто проигнорируют их, даже если картинка несет смысловую нагрузку.
    *   Пустые ссылки (`href=""`) и неинтерактивные иконки без `aria-label`.
    *   Кнопки и ссылки не имеют визуальных стейтов `:focus-visible`, что делает навигацию с клавиатуры невозможной.
*   **Ошибки контента (Typos):** В тексте куча опечаток, которые портят впечатление: *"Alcohole by valume"* (Alcohol by volume), *"MounthFeels"* (Mouthfeel), *"bottel"* (bottle).
*   **Вложенность и дублирование:** Блоки вроде `#page2_part2` содержат дублирующуюся разметку, которую можно было бы вынести в переиспользуемые компоненты (карточки характеристик пива).

### 2. CSS (Архитектура, специфичность, костыли)
*   **Засилье ID-селекторов:** Весь CSS построен на ID (`#main`, `#page1`, `#page4_color_image1_text`). Это грубейший антипаттерн. ID имеют огромную специфичность, их невозможно переиспользовать, а переопределение стилей превращается в ад.
*   **«Магические числа» и хардкод:** В коде полно значений вроде `width: 94.2vw`, `left: 5vw`, `margin: 26vh 0 0 -6vh`. Верстка держится «на соплях» и сломается при малейшем изменении пропорций экрана.
*   **Отсутствие дизайн-токенов (CSS-переменных):** Цвета (например, `rgb(199, 29, 29)`), шрифты и отступы захардкожены. Нет единого источника правды (`:root`).
*   **Дублирование кода (DRY):** Стили для `#page4_color_image1_text`, `#page4_color_image2_text` и `#page4_color_image3_text` почти идентичны. Это нужно объединять в один класс-компонент (например, `.flavor-card`).
*   **Code Smells:** Использование `!important` (`overflow-y: hidden !important;`) — признак бессилия перед специфичностью. Скрытие скроллбара (`::-webkit-scrollbar { display: none; }`) — плохой UX.

### 3. Адаптивность и Кроссбраузерность (RWD)
*   **Полное отсутствие медиазапросов (`@media`):** В проекте нет адаптива. Автор попытался сделать его «резиновым» за счет `vw` и `vh`, но на мобильных устройствах (где `vw` маленький, а `vh` большой) этот макет просто разорвет на части. Текст станет нечитаемым, а позиционирование картинок улетит за пределы экрана.
*   **Desktop-only подход:** Код написан исключительно под один размер монитора автора. Мы будем переписывать по принципу **Mobile-first**.

### 4. JavaScript (Логика, производительность, анимации)
*   **Ошибки в коде (Typos):** В GSAP-анимациях написано `xPrecent: 200` вместо правильного `xPercent: 200`. Из-за этого часть анимаций просто не работает.
*   **Ошибки селекторов:** В JS вызывается `gsap.from("#page6_part6>button"...)`, но в HTML такого ID нет (там `#page6_part2`).
*   **Устаревшие библиотеки:** Используется `locomotive-scroll` версии 3.5.4 (очень старая). Сейчас стандартом де-факто для smooth-скролла является **Lenis** (он производительнее и лучше работает с нативным скроллом) или Locomotive v4.
*   **Глобальная область видимости:** Весь код лежит в глобальном скоупе. Нет модульности, нет разделения на инициализацию скролла и настройку анимаций.
*   **Производительность анимаций:** Анимации применяются к свойствам, вызывающим reflow/repaint (например, масштабирование через `scale` без `will-change: transform`).

## ШАГ 2 — Целевая архитектура

### 1. Идеальная структура проекта (Component-Driven)
```text
📦 src/
 ┣ 📂 assets/              # Шрифты, иконки, оптимизированные изображения (WebP/AVIF)
 ┣ 📂 js/
 ┃ ┣ 📂 modules/           # Изолированные JS-модули
 ┃ ┃ ┣ 📜 scroll.js        # Настройка smooth-скролла (Lenis)
 ┃ ┃ ┗ 📜 animations.js    # Логика GSAP и ScrollTrigger
 ┃ ┗ 📜 main.js            # Точка входа (инициализация модулей)
 ┣ 📂 scss/
 ┃ ┣ 📂 abstracts/         # Токены, миксины, функции (не компилируются в CSS напрямую)
 ┃ ┃ ┣ 📜 _variables.scss  # CSS-переменные (Design Tokens)
 ┃ ┃ ┗ 📜 _mixins.scss     # Миксины для медиазапросов и типографики
 ┃ ┣ 📂 base/              # Базовые стили
 ┃ ┃ ┣ 📜 _reset.scss      # Современный CSS-сброс (Modern CSS Reset)
 ┃ ┃ ┗ 📜 _typography.scss # Глобальные правила шрифтов
 ┃ ┣ 📂 components/        # Независимые UI-компоненты (БЭМ-блоки)
 ┃ ┃ ┣ 📜 _button.scss     # .btn
 ┃ ┃ ┣ 📜 _card.scss       # .flavor-card, .feature-card
 ┃ ┃ ┗ 📜 _navigation.scss # .nav
 ┃ ┣ 📂 layout/            # Крупные секции (макро-лейаут)
 ┃ ┃ ┣ 📜 _header.scss     # .header
 ┃ ┃ ┣ 📜 _hero.scss       # .hero (бывший #page1)
 ┃ ┃ ┗ 📜 _section.scss    # Базовые отступы секций
 ┃ ┗ 📜 style.scss         # Главный файл (собирает все импорты)
 ┗ 📜 index.html           # Семантичная разметка
```

### 2. БЭМ-конвенция (Strict BEM)
Мы полностью избавляемся от ID в стилях. ID останутся только в HTML для JS-хуков и якорных ссылок (с префиксом `js-`, например `id="js-hero-bottle"`).

### 3. Система Дизайн-Токенов (Design Tokens)
Мы вынесем все хардкод-значения в единый источник правды — CSS-переменные в корне `:root`. Используем `clamp()` для Fluid Typography & Spacing.
